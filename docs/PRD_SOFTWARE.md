## 3. 产品定位

第二人生是一款互动视觉小说体验平台。用户可以进入不同人生故事，体验他人的秘密、选择、关系、代价和结局。

第一版开放《雨夜来电》，它属于悬疑人生故事。产品不再局限于单一悬疑内容，未来可扩展到悬疑人生、都市人生、爱情人生、职场人生、重启人生、逆转人生、平行人生等方向。

## 4. MVP 范围

1. 原生 HTML + CSS + JavaScript 前端 Demo。
2. 无后端、无登录、无支付、无后台管理。
3. 支持启动页、人生档案首页、人生档案详情页、剧情页。
4. 支持剧情推进、有效选择、人物关系值、线索库、真相进度、存档/读档、历史记录、最终推理和多结局。
5. 使用 localStorage 保存本地进度。
6. 当前故事《雨夜来电》采用 6 个长章节、6 条正式关键线索。

## 6. 沉浸式体验节奏原则

第一版重构目标是让剧情沉浸优先于系统反馈。反馈系统应服务悬疑推进，不能压过故事本身。

### 6.1 不频繁打断用户

1. 不要每获得一点小信息都弹窗。
2. 不要频繁章节结算。
3. 不要让线索、成就、人物关系等反馈系统压过剧情本身。
4. 玩家阅读连续剧情时，应优先保持沉浸。

### 6.2 反馈分级

| 信息等级 | 反馈方式 |
| --- | --- |
| 普通信息 | 只在正文、对话、动作中体现，不弹窗。 |
| 关键线索 | 轻量线索卡，可关闭后继续。 |
| 大章节结束 | 展示章节成果。 |
| 重大反转 | 展示里程碑。 |
| 结局 | 集中展示完整复盘。 |

### 6.3 弹窗节奏

1. 线索弹窗只对正式关键线索触发。
2. 里程碑最多 4 个。
3. 章节结算只在 6 个大章节结束时触发。
4. 成就只用 toast，不强制弹窗。
5. 人物关系变化只 toast 或人物面板展示，不强制弹窗。
6. 场景热点反馈只 toast，不弹大窗。

## 7. 章节节奏调整

故事结构从 12 个短章节调整为 6 个长章节。

| 章节 | 名称 | 章节目标 |
| --- | --- | --- |
| 第 1 章 | 雨夜来电 | 开场强钩子，完成死者来电、门铃、门外女人和周屿消息。 |
| 第 2 章 | 门外的许知晚 | 围绕“她到底是谁”制造身份疑云和信任压力。 |
| 第 3 章 | 三年前的裂缝 | 从怪事推进到旧案疑点，查出灰色借贷与周屿离城。 |
| 第 4 章 | 最后一张合照 | 让玩家参与照片检查，发现背景人影。 |
| 第 5 章 | 旧手机的声音 | 解释伪灵异机制，并把现实威胁推到最高。 |
| 第 6 章 | 无人接听 | 完成最终推理、最终选择和 A/B/C/D 结局报告。 |

每章 18-25 个节点，总节点建议 120-150 个。每章至少 3-5 个有效选择；开头有悬念，中段有压力升级，结尾有强钩子。

## 8. 线索库减负原则

正式线索从过多条目压缩为 6 条关键线索。线索库只展示真正影响最终推理和结局路径的证据。

| clueId | 名称 | 分类 | 用途 |
| --- | --- | --- | --- |
| clue_dead_call | 死者来电 | 通话 | 开启事件，证明“已故者号码来电”发生过。 |
| clue_sister_mark | 许知晚身份细节 | 人物 | 支撑许知晚并非随便冒名。 |
| clue_gray_loan | 灰色借贷 | 旧案 | 指向许知夏死亡动机。 |
| clue_zhou_left | 周屿离城 | 人物 | 指向周屿时间线异常。 |
| clue_photo_background | 照片背景人影 | 照片 | 关键物证，证明周屿出现在不该出现的位置。 |
| clue_timed_voice | 旧手机语音触发记录 | 通话 | 解释死者来电机制。 |

以下内容不再作为正式线索进入线索库：警告句、门外女人、旧手机本身、最后合照本身、房东证词、旧聊天记录、周屿异常短信、合照背面字迹、旧手机开机时间。

这些信息仍然可以存在于剧情、对话、动作、证据细节、结局路径说明中，但不触发线索卡。

真相进度改为：

```text
真相进度：x / 6
```

## 9. 未解锁内容展示规范

禁止大量展示问号堆叠、未知线索和空白占位。

线索库默认优先展示已获得线索。未获得线索折叠到“未归档”区域，使用氛围化表达：

| 状态 | 展示文案 |
| --- | --- |
| 未归档线索 | 这条记录还藏在雨声里。 |
| 身份未确认 | 你还没有足够证据判断这个人是否可信。 |
| 档案封存中 | 这段人生还没有向你打开。 |
| 尚未抵达 | 另一种结果，仍停在未发生的选择之后。 |

未获得线索可以显示“未归档线索 01 / 02”，但不要暴露具体线索名，也不要让未获得线索数量压迫玩家。

## 10. 有效选择设计规范

每章至少 3-5 个有效选择。每个选择必须至少影响以下一项：

1. 人物关系值。
2. flags。
3. 是否获得关键线索。
4. 结局路径说明。
5. 章节总结。
6. 玩家心理倾向。

不要加入“看左边 / 看右边”“继续观察 / 再观察”等没有后果的假选择。

推荐选择类型：

| 类型 | 示例 |
| --- | --- |
| 信任选择 | 隔着门继续问许知晚；让她先进门；要求她发证明照片。 |
| 行动选择 | 给陈妍发消息；回拨旧号码；查看猫眼；录下周屿电话；先备份照片。 |
| 推理选择 | 先查旧新闻；先看聊天记录；先放大照片背景；先核验旧手机时间。 |
| 证据选择 | 备份照片；只发压缩图；保留原件；交给许知晚；删除证据。 |
| 对抗选择 | 接周屿电话；不接周屿电话；质问周屿；假装不知道照片；诱导周屿说漏嘴。 |

## 11. 对话型视觉小说体验原则

剧本不是普通小说，不应大段旁白堆叙述。

1. 旁白不超过 35%。
2. 人物对话、动作、当前冲突、选择内容至少 65%。
3. 不要用旁白直接解释真相。
4. 真相要通过对话、行动、细节、线索拼出来。
5. 角色说话要短，有隐瞒，有试探，有情绪。
6. 旁白只负责氛围、动作和必要衔接。

错误写法：

```text
旁白：周屿其实很可疑，因为他三年前离开城市。
```

更好写法：

```text
陈妍：你猜他什么时候离职的？
林舟：案发后？
陈妍：不。新闻出来前一天。
门外的许知晚忽然安静了。
周屿的新消息跳出来：别再查了。
```

## 12. 人物关系系统

保留人物关系系统，但降低打断。

| id | 展示 | 含义 |
| --- | --- | --- |
| trust_zhuwan | 许知晚：信任 | 林舟对许知晚的信任与许知晚配合程度。 |
| support_chenyan | 陈妍：协助 | 陈妍对林舟的现实支撑和证据协助。 |
| suspicion_zhou | 周屿：警觉 | 周屿对林舟行动的警觉程度和施压强度。 |
| courage_linzou | 林舟：直面 | 林舟面对旧案、愧疚和真相的程度。 |

关系变化可以 toast，人物面板可随时查看，关系变化进入章节总结和结局报告。关系系统服务剧情，不抢剧情。

## 14. 结局判断规则

A 结局不要求全部 6 条线索，但必须要求关键证据链完整：

1. 已获得 `clue_gray_loan`。
2. 已获得 `clue_zhou_left`。
3. 已获得 `clue_photo_background`。
4. 已获得 `clue_timed_voice`。
5. `backed_up_photo = true`。
6. `chose_reopen_case = true`。
7. `deductionScore >= 4`。

C 结局优先：`deleted_evidence = true` 且 `backed_up_photo !== true`。

B 结局：`gave_original_photo = true`，且身份核验、证据备份或来电机制存在缺口。

D 结局：不满足 A/B/C，或证据链不足、推理不足、没有推动重启旧案。

## 15. 结局报告加强

结局报告必须展示：

1. 结局名称。
2. 结局类型。
3. 真相进度。
4. 推理成绩。
5. 关键选择。
6. 人物关系值。
7. 结局评价。
8. 你如何走到这个结局。

“你如何走到这个结局”根据 state 自动生成路径链。

A 结局路径示例：

1. 你查到了灰色借贷，确认许知夏生前正在自救。
2. 你发现了周屿案发后离城。
3. 你在照片背景里发现了周屿。
4. 你理解了死者来电不是灵异，而是旧手机语音触发。
5. 你备份了关键证据。
6. 你最终推理达到 4/5。
7. 你选择重启旧案。
8. 因此进入：真相重启。
