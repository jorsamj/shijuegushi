# 《迷雾剧本馆》软件产品 PRD

## 1. 产品背景

视觉小说适合承载悬疑推理内容，但传统单篇 Demo 往往只呈现一个故事，缺少“剧本馆”式的连续内容承载能力。第一版需要先验证用户是否愿意通过剧本馆首页、系列详情页和剧情播放页逐层进入故事，并在本地完成选择、线索收集、推理判断和多结局体验。

《迷雾剧本馆》的第一阶段不追求完整商业化能力，而是建立可扩展的前端 Demo 外壳，让后续多个悬疑系列和多本剧本可以用统一结构接入。

## 2. 产品定位

迷雾剧本馆是一个悬疑推理视觉小说剧本馆。用户进入后，先看到剧本馆外壳，再选择悬疑系列，再选择某一本剧本进入游玩。每个系列下面可以包含多本剧本，第一版只开放“凌晨三点系列”的第 1 本《凌晨三点的监控》，其他剧本和系列以“未完待续”状态展示。

## 3. 目标用户

| 用户类型 | 特征 | 核心需求 |
| --- | --- | --- |
| 悬疑推理爱好者 | 喜欢案件、线索、反转和多结局 | 希望在短时间内完成一段可推理的互动故事 |
| 视觉小说玩家 | 熟悉剧情推进、选项分支、历史记录 | 希望体验清晰的剧情播放和可回看的文本记录 |
| 前端 Demo 体验者 | 关注页面结构、交互完整度和可扩展性 | 希望看到无需后端也可运行的完整产品雏形 |

## 4. 核心卖点

1. 剧本馆式内容入口：用系列和剧本组织悬疑内容，而不是单篇孤立故事。
2. 悬疑推理闭环：剧情推进、选择分支、线索收集、最终推理和结局判断形成完整体验。
3. 本地存档能力：通过 localStorage 支持进度保存、读档和历史记录。
4. 多剧本可扩展：剧本、章节、节点、线索和结局均按数据结构组织，方便后续扩展。

## 5. MVP 范围

| 优先级 | 范围 | 说明 |
| --- | --- | --- |
| P0 | 前端 Demo | 使用 HTML + CSS + JavaScript，不使用前端框架 |
| P0 | 剧本馆首页 | 展示产品名称、系列入口、开放状态 |
| P0 | 系列详情页 | 展示系列介绍和剧本列表 |
| P0 | 游戏剧情页 | 支持文本推进、选项分支、线索收集和结局判断 |
| P0 | localStorage | 保存当前进度、存档槽、已收集线索、历史记录 |
| P1 | 弹窗系统 | 支持线索、存档/读档、历史记录、未完待续提示 |
| P1 | 占位图 | 不接入真实图片，只使用占位图或占位区域 |

## 6. 非 MVP 范围

1. 不需要后端服务。
2. 不需要用户登录。
3. 不需要支付、订阅或会员系统。
4. 不需要后台管理。
5. 不需要真实图片、音频、视频或素材生产。
6. 不需要前端框架。
7. 不需要移动端原生 App。
8. 不需要云端同步存档。

## 8. 页面流程

1. 用户打开 Demo，进入启动页。
2. 用户点击进入，来到剧本馆首页。
3. 用户选择“凌晨三点系列”，进入系列详情页。
4. 用户选择《凌晨三点的监控》，进入游戏剧情页。
5. 用户点击或按键推进剧情。
6. 遇到选择节点时，用户选择行动方向。
7. 系统根据选择更新剧情节点、线索、历史记录和进度。
8. 用户可随时打开线索、存档/读档或历史记录弹窗。
9. 用户进入最终推理节点后，系统根据线索和选择触发结局。
10. 用户点击未开放系列或剧本时，展示“未完待续”提示弹窗。

## 9. 页面功能说明

| 页面/弹窗 | 主要功能 | MVP 优先级 |
| --- | --- | --- |
| 启动页 | 展示产品氛围、进入按钮 | P0 |
| 剧本馆首页 | 展示系列列表和开放状态 | P0 |
| 系列详情页 | 展示剧本列表、剧本状态和入口 | P0 |
| 游戏剧情页 | 播放剧情、处理选项、收集线索、判断结局 | P0 |
| 线索弹窗 | 查看已获得线索和线索说明 | P1 |
| 存档/读档弹窗 | 保存当前进度、读取历史存档 | P1 |
| 历史记录弹窗 | 查看本次游玩已出现文本 | P1 |
| 未完待续提示弹窗 | 提示内容未开放 | P0 |

## 10. 剧本馆首页功能

1. 展示产品名称“迷雾剧本馆”。
2. 展示简短副标题，说明这是悬疑推理视觉小说剧本馆。
3. 展示三个系列：
   - 凌晨三点系列：已开放。
   - 雨夜来电系列：未开放。
   - 旧楼档案系列：未开放。
4. 已开放系列可点击进入系列详情页。
5. 未开放系列点击后弹出“未完待续”提示。
6. 系列卡片需包含系列名称、状态、简短简介和剧本数量占位。

## 11. 系列详情页功能

1. 展示系列名称“凌晨三点系列”。
2. 展示系列简介和开放状态。
3. 展示剧本列表：
   - 第 1 本：《凌晨三点的监控》，已开放。
   - 第 2 本：《凌晨三点的来电》，未完待续。
   - 第 3 本：《凌晨三点的空房间》，未完待续。
4. 已开放剧本可进入游戏剧情页。
5. 未完待续剧本点击后展示提示弹窗。
6. 支持返回剧本馆首页。

## 12. 游戏剧情页功能

1. 显示当前剧本名称、章节名和当前场景。
2. 显示角色名或旁白标识。
3. 显示剧情文本。
4. 支持点击“继续”或等价操作推进到下一个节点。
5. 遇到选项节点时，展示多个选项按钮。
6. 选择后根据节点配置跳转到不同剧情节点。
7. 获得线索时展示提示，并写入线索系统。
8. 重要节点写入历史记录。
9. 支持打开线索弹窗、存档/读档弹窗、历史记录弹窗。
10. 到达结局节点时展示结局名称、结局说明和返回入口。

## 13. 线索系统设计

| 字段 | 说明 |
| --- | --- |
| clueId | 线索唯一 ID |
| title | 线索名称 |
| description | 线索描述 |
| category | 线索分类，例如监控、现场、证词、系统日志 |
| chapterId | 首次出现章节 |
| nodeId | 首次出现节点 |
| isKey | 是否关键线索 |

线索获得规则：

1. 剧情节点可配置 `gainClues` 字段。
2. 当玩家到达该节点时，系统检查线索是否已存在。
3. 未获得过的线索加入已收集线索列表。
4. 已获得过的线索不重复加入。
5. 关键线索可参与最终结局判断。

## 14. 存档/读档系统设计

第一版建议提供 3 个本地存档槽。每个存档槽保存当前剧本进度和玩家状态。

存档内容包括：

1. 剧本 ID。
2. 当前章节 ID。
3. 当前节点 ID。
4. 已收集线索 ID 列表。
5. 已触发关键选择。
6. 历史记录摘要或完整历史记录。
7. 存档时间。

交互规则：

1. 玩家点击“存档”后选择槽位。
2. 空槽位显示“空存档”。
3. 已有存档槽显示剧本名、章节名、节点摘要和保存时间。
4. 覆盖已有存档前需要二次确认。
5. 玩家点击“读档”后选择槽位并恢复状态。
6. 读取不存在或损坏的存档时展示错误提示，并保持当前进度不变。

## 15. 历史记录设计

历史记录记录玩家在本次游玩中已经看到的重要文本，用于回看剧情和推理信息。

记录内容包括：

| 字段 | 说明 |
| --- | --- |
| recordId | 历史记录 ID |
| nodeId | 来源节点 ID |
| speaker | 说话人或旁白 |
| text | 文本内容 |
| chapterId | 章节 ID |
| createdAt | 记录时间 |

规则：

1. 普通剧情文本进入历史记录。
2. 系统提示类文本可不进入历史记录。
3. 选项文本需要记录玩家选择。
4. 历史记录按出现顺序倒序或正序展示，第一版建议正序。

## 16. 未完待续机制

适用对象：

1. 未开放系列。
2. 未开放剧本。
3. 后续章节占位。

触发规则：

1. 用户点击未开放内容时，不跳转页面。
2. 系统展示统一的未完待续提示弹窗。
3. 弹窗文案包含内容名称和“未完待续”状态。
4. 用户点击确认后关闭弹窗，保留当前位置。

## 18. 剧情节点数据结构建议

```json
{
  "nodeId": "n_001",
  "chapterId": "chapter_01",
  "type": "dialogue",
  "speaker": "林澈",
  "text": "这里放剧情文本摘要或正式文本。",
  "background": "placeholder_temple_night",
  "gainClues": ["clue_lost_7_seconds"],
  "nextNodeId": "n_002",
  "choices": [
    {
      "choiceId": "c_check_log",
      "text": "先检查系统日志",
      "nextNodeId": "n_010",
      "flags": ["checked_log"]
    }
  ],
  "endingId": null
}
```

节点类型建议：

| 类型 | 用途 |
| --- | --- |
| dialogue | 普通剧情文本 |
| choice | 选项分支 |
| clue | 线索获得节点 |
| deduction | 推理判断节点 |
| ending | 结局节点 |

## 19. localStorage 存储字段建议

| Key | 内容 | 说明 |
| --- | --- | --- |
| mist.currentProgress | 当前自动进度 | 保存当前剧本、章节、节点和状态 |
| mist.saveSlots | 手动存档槽 | 保存多个槽位 |
| mist.collectedClues | 已收集线索 | 可与进度合并，也可独立存储 |
| mist.history | 历史记录 | 保存本次或最近一次游玩记录 |
| mist.settings | 本地设置 | 预留字体速度、音量等设置 |

字段命名建议统一使用 `mist.` 前缀，避免与其他 Demo 的 localStorage 字段冲突。

## 20. UI 风格说明

1. 整体风格：悬疑、克制、冷静，有档案馆和夜间监控感。
2. 色彩方向：深色背景、低饱和灰蓝、暗红或警示黄作为少量强调色。
3. 字体方向：标题可偏电影感，正文优先保证可读性。
4. 页面氛围：通过布局、遮罩、占位图和文案营造悬疑感，不依赖真实图片。
5. 信息密度：剧情页优先阅读体验，工具按钮保持清晰可见但不抢戏。

## 21. 交互规则

1. 所有可点击元素需要有明确 hover/active 状态。
2. 未开放内容不允许进入详情或剧情页。
3. 剧情推进时，如果当前节点存在选项，则必须先选择选项，不自动跳过。
4. 线索获得提示不应阻断剧情过久，用户可关闭后继续。
5. 弹窗打开时，背景页面不响应主要操作。
6. 读档会覆盖当前临时进度，执行前需要确认。
7. localStorage 写入失败时，需要给出提示，并允许继续当前会话。

## 22. 第一版验收标准

1. 打开 Demo 后能从启动页进入剧本馆首页。
2. 剧本馆首页展示三个系列及开放状态。
3. 点击“凌晨三点系列”可进入系列详情页。
4. 系列详情页展示三本剧本及开放状态。
5. 点击《凌晨三点的监控》可进入游戏剧情页。
6. 点击未开放系列或剧本时展示“未完待续”弹窗。
7. 游戏剧情页支持至少一条主线剧情推进。
8. 游戏剧情页支持至少一次选项分支。
9. 玩家可获得并查看至少 5 条线索。
10. 玩家可打开历史记录并看到已阅读文本。
11. 玩家可保存进度到 localStorage。
12. 玩家可从 localStorage 读取存档并恢复到对应节点。
13. 最终推理可触发 A/B/C/D 四类结局之一。
14. 刷新页面后，最近进度或存档不丢失。
15. 项目中不包含后端、登录、支付、后台管理或真实素材依赖。

## 23. 后续迭代方向

1. 增加更多剧本和系列。
2. 增加剧情数据编辑规范或轻量编辑器。
3. 增加音效、背景音乐和正式视觉素材。
4. 增加成就、结局图鉴和线索图谱。
5. 增加云端账号与跨设备存档。
6. 增加付费解锁或章节制内容管理。
7. 增加移动端适配和 PWA 支持。

## 24. 风险点

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 剧情数据结构过早复杂化 | 增加开发成本 | MVP 只保留必要字段 |
| localStorage 数据损坏 | 影响读档体验 | 读取时增加校验和异常提示 |
| 线索与结局判断不清晰 | 玩家无法理解推理结果 | 在故事 PRD 中明确线索映射和结局条件 |
| 未开放内容过多 | 用户误以为 Demo 不完整 | 用清晰状态和未完待续弹窗管理预期 |
| 无真实素材导致氛围不足 | 影响第一印象 | 用版式、文案和占位视觉先建立悬疑感 |

## 25. 开发前待确认问题

1. 第一版剧情文本量是否控制在 15-30 分钟体验内？
2. 存档槽数量是否固定为 3 个？
3. 历史记录是否需要跨读档保留，还是按当前进度恢复？
4. 最终推理是一次性选择，还是允许失败后重试？
5. 是否需要提供“重新开始剧本”入口？
6. 是否需要支持键盘快捷键推进剧情？
7. 第一版是否需要响应式移动端适配，还是优先桌面端？
