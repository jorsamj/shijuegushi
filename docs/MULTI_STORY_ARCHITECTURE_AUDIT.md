# 第二人生多故事架构审查

审查日期：2026-07-11

## 结论

当前运行时仍是单故事结构，不能直接接入第二个正式故事。必须先把“故事档案”抽成独立 profile，再复用同一套画面、移动端、音频与弹框引擎。

## 已确认的 P0 硬编码

| 区域 | 当前情况 | 必须改造 |
| --- | --- | --- |
| 当前故事 | `script_rain_call` 和 `ch01_001` 直接写在 `script.js` | 由当前 story profile 提供 `scriptId` 与 `startNodeId` |
| 存档 | `mist.currentProgress`、`mist.saveSlots`、`mist.history` 单键共享 | 改为按 `scriptId` 命名空间保存 |
| 玩法数据 | `CORE_CLUE_IDS`、关系、成就、里程碑和结局报告是雨夜来电专用 | 每个 story profile 维护独立线索、规则、关系、成就和结局解释 |
| 首页与详情 | 首页仅突出一个开放 series；详情默认读取雨夜来电 | 从开放脚本注册表动态渲染两个独立档案 |
| 视觉与音频 | 全局映射只包含雨夜来电场景、人物、道具与音频语义 | 第二故事使用完全独立的场景、人物、道具、线索和音频键空间 |
| QA | 流程、语义和资源检查默认 `ch01_*` | 新增多故事隔离、第二故事流程和跨故事回归检查 |

## 目标结构

每个故事以独立 profile 提供：`scriptId`、起点、章节、节点、线索、规则、关系、结局、结算报告、资产命名空间和状态默认值。通用引擎只读取当前 profile，不复制 `script.js`。

状态键采用 `mist.story.<scriptId>.<kind>`：自动进度、三个存档槽、历史和局部收藏均独立；音量、阅读与安装引导继续使用产品级 `mist.settings`；全局档案馆仅汇总每个故事的完成情况。

## 兼容边界

- `script_rain_call` 保留现有节点 ID、存档内容和视觉路径。
- 第二故事全部使用 `dorm_*` ID，禁止使用 `ch01_*`、`clue_*`、`ending_a` 等雨夜来电 ID。
- 过渡期会读取旧版雨夜来电存档键并迁移到其命名空间，避免已有玩家进度丢失。
